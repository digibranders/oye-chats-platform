OyeChats Platform Documentation

Internal developer documentation for the OyeChats SaaS chatbot platform.

What is OyeChats?

OyeChats is a SaaS chatbot platform where customers sign up, create chatbot instances, upload their knowledge base, and embed an AI chatbot on their website with a single script tag. The chatbot uses RAG (Retrieval-Augmented Generation) to answer visitor questions from the customer's documents.

How It Works

Customer signs up → Creates a bot → Uploads docs / crawls URL → Documents chunked, embedded, stored in pgvector → Copies embed script into their website → Visitors see chat widget → Ask questions → RAG pipeline: hybrid search → LLM generation → Streaming response

Architecture Overview

System architecture, application map, and data flow diagrams.

System Map

oye-chats/ ├── platform/ │ ├── api/ → FastAPI REST API + RAG pipeline (port 8000) │ ├── widget/ → Embeddable chat widget IIFE bundle (port 5173 / 4173) │ ├── admin/ → React admin dashboard SPA (port 5174) │ └── docs/ → This documentation └── landing/ → Next.js marketing site (port 3000)

Application Responsibilities

Backend API (api/)

The central nervous system. Handles business logic, data persistence, and AI orchestration. Built on FastAPI 0.115+ with SQLAlchemy 2.0 ORM, PostgreSQL 16 with pgvector, and LiteLLM for LLM abstraction.

Chat Widget (widget/)

A self-contained IIFE JavaScript bundle (~416KB) that customers embed on their websites. Bundles its own React 19 instance for full isolation. Communicates with the backend via REST + SSE streaming.

Admin Dashboard (admin/)

React SPA for bot management, document uploads, analytics dashboards, and live chat operator tools. Uses React Router 7 and Recharts for data visualization.

Visitor Chat Flow

1. Visitor loads page → Widget script executes, reads data-bot-key 2. Widget fetches config → GET /bots/settings/public (colors, logo, prompt) 3. Visitor asks question → POST /chat/stream with X-Bot-Key header 4. Backend runs RAG: a. Hybrid search → Vector similarity + keyword (TSVECTOR) b. Context assembly → Top chunks + chat history + system prompt c. LLM generation → LiteLLM → streaming SSE response d. BANT extraction → Background sales qualification (if enabled) 5. Response streams back → Widget renders markdown in real-time 6. Message stored → ChatMessage record with Langfuse trace_id

Live Chat Handoff Flow

1. Visitor requests handoff → POST /operators/handoff 2. Session status changes → bot → waiting → live 3. Operator matched → Round-robin from available operators 4. Real-time messaging → WebSocket (/ws) 5. If no operator available → Offline message queued

Observability

Langfuse (LLM Tracing)

Every LLM call is traced via LiteLLM auto-callbacks. Trace IDs are stored on ChatMessage.trace_id for end-to-end debugging. Widget feedback (thumbs up/down) is linked back to traces. Requires LANGFUSE_SECRET_KEY and LANGFUSE_PUBLIC_KEY.

Sentry (Error Tracking)

Runtime errors are captured with automatic endpoint tagging. 10% trace and profile sample rates (configurable). Widget has its own optional Sentry via VITE_SENTRY_DSN.

Chat Routes

POST /chat/stream Send message, receive SSE stream — Auth: X-Bot-Key

Request Body

{
  "message": "What are your pricing plans?",
  "session_id": "sess_abc123"  // optional
}

SSE Response Protocol

METADATA:{"sources": ["pricing.pdf"], "session_id": "sess_abc123"}
Here is information about our pricing...
We offer three tiers:
...
FINAL_METADATA:{"message_id": 456, "trace_id": "tr_xyz789"}

Lines prefixed with METADATA: contain JSON with sources and session ID. FINAL_METADATA: contains the message ID and Langfuse trace ID. All other lines are response text chunks.

POST /chat/feedback/{message_id} Submit feedback — Auth: X-Bot-Key 
{ "feedback": 1 }  // 1 = up, -1 = down, 0 = reset
GET /chat/history/{session_id} Get session messages — Auth: X-Bot-Key

Returns an array of messages with id, role, content, feedback, trace_id, created_at.

RAG Pipeline & Ingestion

Document processing, embedding, hybrid search, and LLM generation.

Ingestion Pipeline

Upload / Crawl ↓ Extract text (PDF → pypdf, DOCX → python-docx, TXT → direct) ↓ Clean text (normalize whitespace, fix encoding) ↓ Hash (SHA-256) → skip if duplicate ↓ Chunk (recursive split, 2000 chars, 300 overlap) ↓ Embed (OpenAI text-embedding-3-small → 1536-dim vectors) ↓ Store (PostgreSQL + pgvector + TSVECTOR)

Chunking Configuration

Chunk overlap300 charactersCHUNK_OVERLAP

Query Pipeline

User Question ↓ Hybrid Search: ├── Vector: embed question → ANN search over bot documents └── Keyword: tsquery → full-text search over TSVECTOR column ↓ Reciprocal Rank Fusion (merge + re-rank) ↓ Context Assembly: ├── System prompt (from bot config) ├── Top retrieved chunks (with source attribution) ├── Chat history (recent messages) └── BANT instructions (if enabled) ↓ LLM Generation (LiteLLM → streaming SSE) ↓ Background: BANT extraction (fire-and-forget) ↓ Store response + trace_id

Hybrid Search: Reciprocal Rank Fusion

The reciprocal_rank_fusion() function merges vector similarity results with keyword search results. Each document gets a combined score that balances semantic relevance with lexical precision. This ensures the system catches both meaning-based matches and exact keyword hits.

BANT Qualification

When bant_enabled = true on a bot, the system subtly qualifies visitors for sales readiness:

DimensionWhat It Tracks
BudgetHas the visitor mentioned pricing, cost, or budget?
AuthorityAre they a decision-maker or influencer?
NeedWhat problem are they solving?
TimelineWhen do they need a solution?

BANT prompts are woven naturally into helpful answers — one question per turn, only when buying signals are detected. When all four fields are populated, an email notification is sent (if configured).

Widget Embedding Guide

How the embeddable chat widget works, build process, and integration.

Production Embed

Add this single line before the closing </body> tag:

<script src="https://cdn.oyechats.com/oyechats-widget.js" data-bot-key="bot-xxx"></script>

How It Works

The widget (oyechats-widget.js) is a ~416KB IIFE bundle that:

StepAction
1Finds its <script> tag, reads data-bot-key
2Sets window.OYECHATS_BOT_KEY globally
3Auto-injects sibling CSS (oyechats-widget.css)
4Creates <div id="oyechats-widget-root">
5Renders its own React 19 app (isolated from host)
6Communicates via REST + SSE with X-Bot-Key header

Build Process

cd widget
npm run build       # → dist/oyechats-widget.js + oyechats-widget.css

Testing Embeds Locally

cd widget
npm run build
npx vite preview --port 4173

Then embed on your test page:

<script src="http://localhost:4173/oyechats-widget.js" data-bot-key="bot-xxx"></script>
The Vite dev server (localhost:5173) cannot be embedded on external sites. Always use the preview build for embed testing.

Isolation & Compatibility

The widget is designed for zero conflicts with host pages. It bundles its own React instance, scopes all DOM and CSS to #oyechats-widget-root, and only sets two window globals. Works on Next.js, WordPress, Webflow, Shopify, plain HTML — anything with a <body> tag.
