1. The core mental model

Everything in Razorpay billing flows from one idea:

Razorpay charges against pre-created objects referenced by ID — not arbitrary amounts. A recurring charge needs a Plan (plan_id). A discount can be an Offer (offer_id) — or, better for us, a discounted Plan. The only place you can pass a free-form amount is a one-time Order (used for top-ups).

This single constraint explains every design decision below: why prices are fixed per currency, why discounts are modelled as discounted plans, and why seats and bots are structured the way they are.

ProductRazorpay primitiveAmount is…Created whereAPI-creatable?
Subscription (plan)plan_id + mandateFixedDashboard or API✅ Yes
Discount (offer)offer_idFixed % or flatDashboard only❌ No
Top-up (credits)orderArbitrary per callCode / DB✅ Yes

That last column is decisive for discounts: offers can't be scripted, plans can. See §8.

2. Currency strategy (USD ↔ INR)

Razorpay subscriptions can only be denominated in INR. We still want to show USD to international visitors. The resolution is a dual-currency fixed model — the same approach Stripe documents and that Netflix / Spotify / Notion use in India.

USD $19 / $49 — the anchor & headline. Fixed. What we advertise & price from.
INR ₹1,799 / ₹4,599 — the actual Razorpay charge. Fixed. Set from the USD anchor.
We never convert currency live for display or charging. Live conversion (e.g. ₹1,799 ÷ today's rate) produces ugly decimals ($18.74), an unstable price that changes every visit, and — for subscriptions — is impossible anyway because the UPI mandate locks a fixed ₹ amount at signup.

Why floating the per-cycle charge is impossible

  • UPI Autopay mandates are fixed-amount — the customer approves a specific ₹ figure; you cannot silently change it month to month.
  • You can only edit a live subscription's amount if it was authorised by card — explicitly not UPI/e-mandate (our primary rail).
  • RBI requires a 24-hour pre-debit notification stating the exact amount before each charge — a different ₹ every month is erratic and dispute-prone.

How exchange-rate moves are handled

Deliberately, not automatically. Day-to-day FX is ignored. When the rupee moves materially (a quarterly review, or a >5% drift), we re-price: mint new Razorpay plans at new ₹ amounts, update the IDs in the DB, and new signups get the new price. Existing customers stay grandfathered on their old plan — standard SaaS practice and good for goodwill. Current INR prices were set at ₹94.67 / $1 (verified 25 Jun 2026).

Where true live-USD will happen later

When INTL_PAYMENTS_ENABLED is switched on, non-Indian customers pay a real $19 via Stripe (which charges USD directly). Indian customers always stay on stable INR via Razorpay.

Top-ups can be scoped to a single bot's isolated credit ledger (per-bot billing) or the shared account pool. Referral discounts intentionally do not apply to top-ups — the affiliate reward tracks recurring revenue only. A one-time $19 top-up grants 2,000 credits vs the $19/mo Starter plan's 3,000/mo — subscribing is intentionally better value.

5. Additional bots reuses plans

A customer's 2nd+ bot is funded by its own subscription created against the same Starter/Standard plan, tagged purpose=per_bot_subscription. The credits land in that bot's isolated ledger. No new dashboard plan required — it reuses Plans 1–4.

6. Operator seats needs fix

Each tier includes operator seats (Starter 1, Standard 2). Extra seats cost ₹499/mo each.

Current bug: seat charges are wrong. The existing code edits the main subscription's quantity, which multiplies the whole plan amount — so adding one seat to Standard would bill ₹4,599 × 2 = ₹9,198, not ₹4,599 + ₹499. The extra_seat_price_cents (₹499) shown in the UI is never actually charged.

The fix: base subscription stays at quantity = 1 (flat plan price); extra seats become a separate add-on subscription on the Extra Seat plan (plan_T5rNFpt3vSkl4R, ₹499) with quantity = number of extra seats. ₹499 × N is correct because that plan's amount is the per-seat price.

7. Affiliates & discounts — overview

There are two independent layers. Keeping them separate is the key to understanding the whole system:

Layer A — Attribution & commission works

A referral code ties a customer to an affiliate; the affiliate earns a commission (affiliate_commission_bps). This is pure internal accounting in our DB — the payment gateway never needs to know about it, and it can be any percentage. Already working.

Layer B — Customer-facing discount to build / broken on Razorpay

The discount the customer actually pays. Today it's only wired for Stripe; on the Razorpay path it is silently dropped — the affiliate gets credited but the customer pays full price. This is what §8 fixes.

The pool split. The super-admin gives each affiliate a pool (e.g. 20%). The affiliate splits it per code into commission (what they keep) + customer discount (what the buyer gets). Example: 20% pool → 8% commission + 12% discount. The sum must be ≤ the pool (enforced in code & DB). The commission half is fully flexible; only the discount half touches the gateway.

8. The discount engine — discounted plans via API

Why not Razorpay Offers

Razorpay's native discount object is an Offer. But offers can only be created from the dashboard — there is no API, and each distinct percentage needs its own hand-made offer. With affiliates free to choose any percentage, that means endless manual work (and possibly one offer per plan, i.e. dozens). Not feasible.

Solution: discounted plans. Plans do have a create API; offers don't. Instead of "base plan + 12% offer," we create a discounted plan at the reduced price on the fly, and reuse it. A lower plan price recurs every cycle by definition — so the discount automatically applies to the first payment and every renewal, which is exactly what affiliate discounts need.
Affiliate sets 12% discount on Standard Monthly (₹4,599)
→ discounted amount = ₹4,599 × 0.88 = ₹4,047
→ look up cache key (Standard, monthly, 12%)
    exists? reuse its plan_id
    not yet? rzp.plan.create(...) once → cache it
→ create the subscription against that discounted plan_id

Plans vs Subscriptions — the distinction that dissolves the "explosion" worry

PlanSubscription
What it isA price template ("Standard −12% = ₹4,047/mo")One customer's recurring mandate
How manyA handful, shared by everyoneOne per customer — always
Reused?✅ Yes, heavily❌ Never (each customer has their own)

Subscriptions = your number of paying customers (10,000 customers = 10,000 subscriptions). That is not proliferation — it is your customer base, identical with or without discounts. The only thing that "accumulates" is plans, and that count is tiny and bounded.

How many discounted plans can ever exist

distinct plans = base plans × billing cycles × distinct discount %s used = 2 (Starter, Standard) × 2 (monthly, annual) × up to 25 (1–25%) = 100 plans MAX. NOT multiplied by affiliates. NOT multiplied by users. Whether you have 10 affiliates or 10,000, 100 customers or 1 million — the discounted-plan count never exceeds this ceiling.

Deduplication — same % always reuses the same plan

Plans are cached by the key (base_plan, billing_cycle, discount_percent). The affiliate's identity is not part of the key — attribution lives on the subscription, separate from the plan.

EventResult
Affiliate A creates a 17% code on Standard MonthlyPlan (Standard, monthly, 17%) created → cached
Affiliate B also creates a 17% code on Standard MonthlySame plan reused — no new plan
A marketing coupon gives 17% on Standard MonthlySame plan reused
Affiliate A creates a 20% codeNew plan (Standard, monthly, 20%) created → cached

50 affiliates · 10,000 customers · discounts actually used = 5%, 10%, 12%, 17%, 20% (5 distinct)

Subscriptions created: 10,000 ← = your customers (normal)
Discounted plans: 5 × 2 base × 2 cycles = 20 plans MAX
20 plan templates serve all 10,000 customers and all 50 affiliates.

Marketing coupons

Same engine. A coupon code resolves to a discount % → a cached discounted plan, exactly like an affiliate code. Time-boxed or first-charge-only promos are the one case where a Razorpay Offer is the better tool (only a couple, created manually as needed).

9. Coupon codes & affiliate rules

Code uniqueness

Codes are globally unique and case-insensitive (CITEXT UNIQUE). Whoever claims ABCD200 first owns it. If another affiliate tries the same code (any casing), the request is rejected with a clean "code already taken." A code is the public key a customer types at checkout — it must resolve to exactly one affiliate and one split, or attribution breaks.

Multiple codes per affiliate

One affiliate can hold many codes (up to their max_active_codes cap). Two codes may carry the identical split (e.g. both 10% commission + 15% discount) — only the code string is unique, not the economics. This is intentional, used for per-channel tracking (JOHN-YT, JOHN-IG).

Editing a code after creation

Affiliates can edit via update_code. The effect on ongoing subscriptions is the crucial part:

EditExisting subscribersNew subscribers
Rename the code string✅ Unaffected — attribution is by row id, not the string. (But old ?ref=OLD links stop working.)Use the new code string
Change customer discount (15%→20%)✅ Grandfathered — locked to their discounted plan & fixed mandate; Razorpay can't change a live recurring amountGet the new discounted plan
Change commission (10%→5%)⚠️ Policy decision — see belowNew rate applies
Deactivate✅ Keep earning / keep their discountBlocked from signing up
Commission edits must snapshot at conversion (industry standard). Commission is computed per payment, not locked in a gateway object. To avoid retroactively changing what affiliates earn on already-referred customers, the commission & discount rate should be snapshotted onto a conversion record when the customer subscribes. Editing the code then affects only new referrals — matching Rewardful / FirstPromoter / PartnerStack. Discounts are already grandfathered automatically by the fixed-plan model.

Recommended guards (industry standard)

  • Reserved / blocklist — stop affiliates claiming brand or generic codes: OYECHATS, FREE, SALE, ADMIN.
  • No discount stacking — one discount per checkout: affiliate code or marketing coupon, never both.
  • Self-referral block — an affiliate can't use their own code on their own subscription.
  • Audit log — every split/commission change recorded (who, when, old → new).

What the system already gets right ✅

Global uniqueness · case-insensitivity · many-codes-per-affiliate · same-split-allowed · split ≤ pool (DB CHECK + service) · format guard (3–20, [A-Za-z0-9_-]) · deactivate-not-delete · clean "code taken" errors. The foundation is solid; the items above are hardening.

10. Payment flows

Subscription checkout (Razorpay)

1. quote → GET /subscriptions/checkout/quote (currency, amount, methods)
2. resolve discount → code/coupon → discount % → cached discounted plan_id
3. checkout → POST /subscriptions/checkout → create_subscription(plan_id)
4. mandate → Razorpay Checkout opens → customer authorises UPI Autopay / card
5. verify → POST /subscriptions/verify (HMAC signature check)
6. webhook → subscription.activated → create local row + snapshot commission + grant credits

Top-up checkout (Razorpay)

1. initiate → POST /credits/topup → create Order (arbitrary ₹ amount)
2. pay → Razorpay Checkout → UPI / card / netbanking / wallet
3. verify → POST /credits/topup/verify (HMAC)
4. webhook → payment.captured → grant credits to the right ledger
Security: webhooks & callbacks are HMAC-verified. Every payment callback and inbound webhook is verified against the raw body with the webhook secret, and webhook events are idempotent (keyed on the Razorpay event ID) so retries never double-grant credits.

14. International payments (Stripe)

How a USA (or any non-Indian) customer checks out — and why Razorpay cannot do it.

Committed architecture: dual-provider, both fully implemented. Razorpay = primary (India, INR) · Stripe = secondary (international, USD). Both rails are built and wired now. Stripe API keys are added later — the system must run cleanly without them and light up the moment they are set, no code change. Razorpay subscriptions are live today; international goes live when Stripe keys land.

The rule

Geo-route the gateway AND the currency. A non-Indian customer must never see INR. Indian visitor → Razorpay → INR → ₹1,799. Non-Indian visitor → Stripe → USD → $19. Each sees their own currency on a gateway that handles it natively.

Non-negotiable: Indian users never see Stripe — no errors, no warnings, no references

Stripe being unconfigured must be invisible on the Indian / Razorpay path. An Indian customer's entire journey runs on Razorpay and must never touch, mention, or fail because of Stripe — whether Stripe keys are present or not.

Concrete guarantees the implementation must hold:

  • Geo decides the provider first. Country resolves before any gateway code runs. IN → Razorpay only — the Indian flow never imports, calls, or awaits Stripe.
  • No boot / runtime coupling. The API already boots fine with no Stripe keys (STRIPE_ENABLED = bool(STRIPE_SECRET_KEY), lazy import). A missing Stripe key must never raise on the Razorpay path or in Razorpay webhooks.
  • Graceful dormancy for international. While Stripe keys are unset, non-Indian visitors get the existing clean "contact sales" fallback — never a 500, never a raw "Stripe not configured" error.
  • Zero Stripe in the Indian UI. No "Powered by Stripe", no Stripe logo, no Stripe redirect, no Stripe string in any error shown to an Indian user. Stripe branding/redirect appears only on the USD path.
  • No console noise. No Stripe-related console errors or network calls on the Indian checkout.
  • Independent webhooks. Razorpay and Stripe webhook handlers are separate; a missing STRIPE_WEBHOOK_SECRET cannot break Razorpay webhook processing.

Net effect: turning Stripe on is purely additive. Until the keys exist, the product behaves exactly as a Razorpay-only, India-only platform — with the international path sitting dormant behind the contact-sales gate.

What a non-Indian customer sees today

Currently they are blocked, not mis-charged: resolve_country() returns e.g. US → checkout returns 402 with a "contact sales" CTA (INTL_PAYMENTS_ENABLED = false gate). They see $19 on the pricing page, then a sales CTA — they cannot pay yet. The gate exists precisely so nobody is routed to a ₹ page.
