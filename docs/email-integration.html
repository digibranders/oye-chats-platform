Sender Identity

Hybrid branding — no DNS setup required for customers

All emails are sent from OyeChats infrastructure (notifications@oyechats.com) but displayed to recipients as coming from the customer's bot. Replies automatically route to the customer's Reply-To address.

From: "Acme Support via OyeChats" <notifications@oyechats.com>
Reply-To: support@acme.com
To: visitor@example.com
Subject: Chat Transcript — Acme Support
ℹ️
Why this approach? Customers don't need to verify a custom domain or configure DKIM/SPF records. The hybrid sender pattern (used by Intercom, Zendesk, Mailchimp) lets replies land in the customer's inbox while ensuring emails pass spam filters from OyeChats' verified sending domain.
✅
Reply-To is set to bot.reply_to_email (configured by the customer in Admin → Integrations → Email). If not configured, the Reply-To header is omitted and replies go to notifications@oyechats.com.

How the sender name is built (backend)

def _branded_sender_name(bot_name: str) -> str:
    """Returns '{bot_name} via OyeChats'"""
    name = (bot_name or "OyeChats").strip()
    return f"{name} via OyeChats"

# Passed to _send_brevo_email() which sets:
payload["sender"] = {"email": "notifications@oyechats.com", "name": sender_name}
if reply_to:
    payload["replyTo"] = {"email": reply_to}

Visitor opens widget menu

2

Widget sends { session_id, recipient_email } with X-Bot-Key header. Rate limited to 3 requests/minute per bot key (SlowAPI).

Confirms the session_id belongs to this bot. Fetches all ChatMessage rows for the session in chronological order.

6

Renders a chat-bubble HTML template: user messages right-aligned (blue), bot messages left-aligned (gray). Timestamps and role labels included. Calls Brevo API asynchronously.

✓

On success: checkmark icon + "Transcript sent to {email}". On error: message with retry option. Modal dismisses automatically after 3 seconds on success.

Recipient Routing

Four-step fallback chain: get_notification_recipients(bot, event_type)

Every team notification uses this resolver. It allows per-event overrides (e.g., route qualified leads to sales, handoff requests to support) while maintaining a global default as fallback.

1
notification_emails[event_type]
E.g. notification_emails["qualified_lead"] → dedicated sales recipients
2
notification_emails["default"]
Global recipients for all events with no override configured
3
notification_email (legacy, comma-split)
Old single-email field. Comma-split for backward compatibility with pre-migration bots.
∅
Empty list — no email sent
No recipients configured for this bot. Notification silently skipped.

JSONB Schema — notification_emails

{
  "default": ["team@acme.com", "manager@acme.com"],
  "qualified_lead": ["sales@acme.com"],         // optional override
  "handoff_request": ["support@acme.com"],       // optional override
  "offline_message": []                            // empty = falls back to default
}
ℹ️
Supported event type keys: qualified_lead, handoff_request, offline_message. The default key is always the global fallback.

🏆 Lead Qualification

Lead Qualification System

How OyeChats scores, tiers, and qualifies visitors through conversational AI — covering all four frameworks, scoring math, tier thresholds, decay, and downstream actions.

🧠

How It Works — Overview

Each visitor conversation is silently scored across multiple dimensions as they chat

OyeChats embeds an LLM-powered qualification engine inside every chatbot. As the visitor talks, the system extracts signals for each dimension (e.g., Budget, Need, Timeline), assigns rubric scores, computes a composite 0–100 score, and classifies the visitor into a tier. This happens in two modes: inline (background signals during normal chat) and SDR (a dedicated qualification conversation).

🔀 End-to-End Qualification Flow
1

Visitor sends a message

Widget sends to either POST /chat (RAG mode) or POST /chat/sdr (SDR mode).

2

LLM extracts dimension signals

Background extraction call after each response. Parses free-text to identify which BANT/framework dimensions are present and at what level.

3

Rubric score assigned per dimension

Each dimension has 4–5 labeled options with discrete point values. E.g., "Urgent need" = 20 pts, "Just browsing" = 5 pts.

4

Composite score computed (0–100)

Weighted average across all enabled dimensions, normalized by each dimension's max possible score.

5

Tier classified

Score → unqualified / MQL / SAL / SQL based on configurable thresholds.

✓

Downstream actions fire on tier upgrade

SQL tier: team notification email, Calendly booking CTA in widget. All tiers: visible in Admin → Leads with full dimension breakdown.

📊

Tiers & Scoring

Composite 0–100 score → 4 tiers with configurable thresholds

🎯 Tier Thresholds (default)
∅
unqualified
Score 0–29 — no meaningful signals yet
M
MQL — Marketing Qualified
Score ≥ 30 — shows intent, needs nurturing
S
SAL — Sales Accepted
Score ≥ 55 — sales-ready, assign to rep
★
SQL — Sales Qualified
Score ≥ 75 — hot lead, triggers email + Calendly CTA
🔢 Composite Score Formula

For each enabled dimension:

normalized = (raw_score / max_possible) * 100
weighted   = normalized × (dim_weight / total_weight)
composite  = Σ weighted   (capped 0–100)

Example (BANT, all 4 dims equal weight):

  • Need: 20/25 → 80 × 0.25 = 20
  • Timeline: 15/25 → 60 × 0.25 = 15
  • Authority: 10/25 → 40 × 0.25 = 10
  • Budget: 5/25 → 20 × 0.25 = 5
  • Composite = 50 → SAL tier
ℹ️
Thresholds are per-bot configurable via bant_config.thresholds (stored as JSONB on the Bot model). The default {"mql": 30, "sal": 55, "sql": 75} applies when no override is set. Admins can tune them in the Qualification settings page.

SDR System Prompt Directives

YOUR DIRECTIVES:
1. ANALYZE & UPDATE   — extract new BANT info from each message, carry forward existing
2. CONVERSATIONAL     — identify null fields, ask ONE question to uncover ONE field
3. NO INTERROGATION   — never ask multiple questions, acknowledge before asking
4. TACTFUL PROBING:
   - Authority: frame around "team involvement" not "are you the boss?"
   - Budget: frame around "tiers" not "how much money do you have?"
5. HANDLING PUSHBACK  — if evasive, pivot to value statement, don't press hard
6. OFF-TOPIC RECOVERY — answer briefly, steer back to business goals
7. THE CLOSE          — all 4 fields populated? Stop probing, propose next steps

Score Decay

Lead scores degrade over time for time-sensitive dimensions

A visitor who said "I need this this month" 6 months ago is no longer an SQL. Decay prevents stale leads from permanently clogging the top of your pipeline. It is display-only — the raw scores in the database are never mutated; decay is computed at read time.

📉 Default Decay Rates
DimensionDecay per 30 daysRationale
Timeline −5 pts / 30 days Most time-sensitive — "this month" becomes "no timeline" quickly
Need −3 pts / 30 days Urgency fades — "critical blocking" may be solved without you
Authority No decay Role/authority doesn't change over time
Budget No decay Budget cycles are long — stays stable unless re-assessed
⚠️
Decay starts after the first full 30-day period since bant_last_updated. It is applied in whole 30-day increments. A visitor updated 45 days ago gets 1 period of decay. One updated 89 days ago also gets 1 period (not 2, since the second period hasn't completed). Configured via bant_config.decay (JSONB per bot).

Decay Logic (lead_service.py)

def apply_display_decay(session, decay_config):
    elapsed = (now_utc - session.bant_last_updated).total_seconds()
    if elapsed <= SECONDS_PER_30_DAYS:
        return raw_scores  # no decay yet
    periods = elapsed // SECONDS_PER_30_DAYS
    need_score     -= periods × need_decay_per_30d
    timeline_score -= periods × timeline_decay_per_30d
    return adjusted_scores  # never below 0
