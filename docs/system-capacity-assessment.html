Scenario: N Clients Crawl Simultaneously

Each /crawl request spawns an independent Python subprocess via asyncio.create_subprocess_exec(). There is no global limit — only per-client locks.

Concurrent ClientsPhase 1 (Chromium)Phase 2 (HTTP)Outcome
1~500 MB~15 MBSafe. Room to spare.
2~1,000 MB~30 MBTight. Swap may activate during 15-30s overlap.
3+~1,500 MB~45 MBMemory gate triggers at 70% (~1.4GB). HTTP 503.
Realistic Capacity
  • 1 concurrent crawl: Safe, no issues
  • 2 concurrent crawls: Works with swap; brief slowdown during Phase 1 overlap
  • 3+ concurrent crawls: Blocked by memory gate (HTTP 503)
  • URLs per crawl: Up to 50 pages, takes 2–8 minutes
  • Throughput: ~6 crawls/hour per client (3/hour × 2 workers with independent counters)
Gap: No Global Subprocess Limit

If 2 clients hit /crawl at the exact same millisecond, both pass _check_memory() before either allocates memory. Both spawn Chromium → 1GB+ spike → possible OOM. The memory gate is checked before the subprocess starts, not after.

Primary Bottleneck: DB Connection Pool (15 max)

Not RAM, not CPU — pool_size=5 + max_overflow=10 = 15 connections is the first wall. At 15 concurrent chatters, new requests queue for up to 30s before timing out.

Note: /chat Is Synchronous

/chat is def chat_endpoint(...) — not async def. Uvicorn runs it in a threadpool (40 threads default per worker). Each thread holds ~80MB and a DB connection for 3–8 seconds. /chat/stream is async and much more efficient.

Key Takeaways

  • Crawling is memory-bound. 1 crawl is safe, 2 is the realistic max. The 70% memory gate protects against OOM.
  • AI chat is DB-pool-bound. 15 concurrent users saturate the pool. Increasing pool_size is the cheapest fix.
  • Live chat is code-limit-bound. Memory cost is trivial (~6 KB/visitor). Raising MAX_QUEUE_SIZE is a 1-line change.
  • Overall, the system is well-optimized for a 2GB droplet. The bottlenecks are configuration limits, not architectural flaws.
